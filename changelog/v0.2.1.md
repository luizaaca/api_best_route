# Release v0.2.1

## Summary

This version completes the next architectural slice of the route optimizer by modularizing heuristic population seeding, introducing persistent caching for geocoding and adjacency segments, formalizing deterministic graph identity, and tightening the dependency boundaries between domain contracts and infrastructure implementations.

## Included Changes

### Genetic algorithm modularization

- extracted GA operators into dedicated infrastructure modules for selection, crossover, mutation, and population generation;
- kept `TSPGeneticAlgorithm` focused on orchestration over injected collaborators;
- introduced `HeuristicPopulationGenerator` alongside `RandomPopulationGenerator` and reduced `HybridPopulationGenerator` to a composition-level generator;
- added heuristic distance strategies for:
  - projected Euclidean distance;
  - adjacency `length`;
  - adjacency `eta`;
  - adjacency `cost`;
- changed the mixed heuristic ordering mode so it diversifies seeded populations with reproducible randomness rather than deterministically preferring convex-hull ordering whenever eligible.

### Domain contract and model refinements

- split domain contracts into dedicated files under `src/domain/interfaces/`;
- split domain models into dedicated files under `src/domain/models/`;
- moved `AdjacencyMatrixMap` from an interface module into `src/domain/models/graph.py`;
- added `graph_id` to `GraphContext` and to the `IRouteCalculator` contract;
- changed `IHeuristicDistanceStrategy` to allow `float | None` when the selected metric is unavailable.

### Heuristic distance correctness

- changed `AdjacencyCostPopulationDistanceStrategy` to return `None` when a segment has no cost instead of silently falling back to ETA;
- updated `HeuristicPopulationGenerator` to fail clearly when a required heuristic metric is missing.

### Persistent caching

- added the `src/infrastructure/caching/` package;
- implemented:
  - `SQLiteGeocodingCache`;
  - `SQLiteAdjacencySegmentCache`;
  - `CachedGeocodingResolver`;
  - `PhotonGeocodingResolver`;
  - `DirectAdjacencyMatrixBuilder`;
  - `CachedAdjacencyMatrixBuilder`;
- updated API and console composition to use the new caching package and persistent caches by default;
- renamed the infrastructure package namespace from `cache` to `caching`.

### Deterministic graph identity

- added deterministic `graph_id` creation in `OSMnxGraphGenerator`;
- based graph identity on normalized bbox coordinates plus the effective graph-selection spec;
- made the effective graph-selection spec use canonicalized `custom_filter` when present, otherwise `network_type`;
- exposed graph identity through `GraphContext` and `RouteCalculator` for typed access by cache-aware builders.

### Typing and safety improvements

- added a minimal internal protocol plus `cast(...)` for Photon geocoding result typing;
- aligned test doubles and fakes with the new graph identity and cost semantics;
- kept explicit interface inheritance on concrete infrastructure implementations.

### Documentation and structure

- updated `README.md` to reflect the current architecture and runtime usage;
- updated `architecture.md` with the current package structure, diagrams, cache flow, and graph identity model;
- added these release notes under `changelog/v0.2.1.md`.

## Validation

Validated in this delivery cycle:

- console execution completed successfully:
  - `python -m console.main`
- the test suite passed after the final fixes for:
  - graph identity flow;
  - heuristic cost semantics;
  - SQLite cleanup behavior on Windows.

## Expected Impact

- cleaner separation between domain contracts and infrastructure implementations;
- easier extension of GA operators and heuristic distance policies;
- safer and reusable cache behavior across runs;
- lower recomputation cost for repeated geocoding and adjacency segment requests;

## Known Limitations

- `weight_type` remains limited to the currently implemented route-calculation strategy;
- adjacency caching is segment-level, not whole-matrix-level;
- the mixed heuristic mode is diversified, but further tuning of heuristic scheduling may still improve convergence;
- Redis-ready abstractions exist conceptually, but SQLite is still the only implemented persistent backend.

## Compatibility

This version preserves the public API shape while changing internal architecture substantially. The main integration-sensitive changes are internal composition, cache namespaces, deterministic graph identity, and stricter heuristic metric semantics.
